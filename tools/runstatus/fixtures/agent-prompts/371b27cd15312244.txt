You are implementing the fix for **TKT-001** — *TUI hangs on ESC*
inside the worktree at `.worktrees/tkt-001`.

The proposing room produced this proposal:

> In `internal/tui/tui.go` inside the `routeKey()` function, extend the existing ESC guard block (currently at line 1251) with a new `ModeAwaitingLLM` branch placed immediately after the `ModeOnPath | ModeOffPath` branch. The new branch must: (1) call `m.inFlightCancel()` if it is non-nil, to cancel the in-flight turn; (2) set `m.mode = ModeMenu`; (3) call `m.menuSystem.Open()`; (4) return `m, nil`. Also remove or update the stale comment on lines 1247–1250 that claims ESC "deliberately does not fire during ModeAwaitingLLM".

**Root cause:** The ESC key handler in `routeKey()` (`internal/tui/tui.go:1252`) only transitions to `ModeMenu` when `m.mode` is `ModeOnPath` or `ModeOffPath`. In `ModeAwaitingLLM` the condition is false, so execution falls through to `m.prompt.Update(msg)`, which delegates to the `charmbracelet/bubbles/textarea` widget. That widget has no ESC binding, so the keystroke is silently consumed. No cancel, no menu, no user feedback.
