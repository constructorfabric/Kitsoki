# Proposal — Cassette oracle-trace fidelity for runstatus snapshots

**Status:** Partially implemented. The deferred oracle event sink
(commit 61f992f, `internal/host/oracle_event_sink.go`) and its wiring
into the test runner (commit 5a9bf56, `internal/testrunner/cassette.go`)
have landed: cassette replays now emit `oracle.<verb>.start/.complete`
trace events that the runstatus UI consumes. Remaining work — if any —
is around fully retiring this proposal into narrative docs
(`docs/tracing/cassettes.md`, `docs/tracing/trace-format.md`); operator should confirm
whether any unfinished aspects justify keeping the proposal before
deletion.

**Tldr.** Host cassettes today substitute *results* (`response.data`)
deterministically, but throw away the rich oracle-call metadata —
prompt, system prompt, response body, tool calls, duration, `call_id`
— that the real engine writes to the SQLite journal as `KindOracleCall`
entries. Consequently, a runstatus snapshot built from a cassette
replay contains zero `oracle.<verb>.complete` events; the runstatus UI's
oracle sub-renderers (`tools/runstatus/src/components/oracle/`) have
nothing to show. This proposal extends cassettes with an optional
`oracle:` block per episode (recorded once against a real LLM, replayed
forever), wires the dispatcher to write a `KindOracleCall` journal
entry on replay, and extends `runstatus.FromHistory` to synthesize
`oracle.<verb>.start/.complete` trace events from that journal stream.
Net effect: bugfix.html — and every future cassette-backed fixture —
gets a faithful oracle trace identical in shape to what `kitsoki run
--trace` produces against a real session. **As of commits 61f992f and
5a9bf56, this net effect is realised in the codebase via a deferred
oracle event sink wired into the test runner.**

---

## 1. Problem

`tools/runstatus/fixtures/bugfix.snapshot.json` is generated by running
`stories/bugfix/flows/happy_human.yaml` through the orchestrator
(`fromflow`) and converting `store.History` to a snapshot via
`internal/runstatus/fromhistory.go`. The flow's `host_handlers:` block
stubs every host call, including `host.oracle.task`, so the real oracle
handlers (`internal/host/oracle_ask.go`, `oracle_decide.go`,
`oracle_extract.go`, `oracle_task.go`, `oracle_converse.go`) never run.
Those handlers are the only code that:

1. Emits `oracle.<verb>.start` / `oracle.<verb>.complete` via
   `slog.InfoContext`.
2. Writes a `KindOracleCall` journal entry (full prompt / response /
   tool_calls / duration / call_id) via `appendOracleCallJournal`.

The runstatus UI's `EventDetail.vue` routes `oracle.<verb>.complete`
events to a rich sub-renderer (`OracleDetail.vue` → `AskDetail`,
`DecideDetail`, `ExtractDetail`, `TaskDetail`, `ConverseDetail`) that
shows the prompt, system prompt, response, tool transcript, and
unified diff for tasks. With zero oracle events in the snapshot, that
entire UI surface is dark.

Verified counts in the current `bugfix.snapshot.json` (131 events):

```
machine.effect:          36
harness.called:          18
harness.dispatched:      18
harness.returned:        18
machine.transition:       9
turn.start:               8
machine.state_exited:     8
machine.state_entered:    8
turn.end:                 8
oracle.*:                 0
```

The same gap exists for `fromsession` against a real SQLite database:
`fromhistory` reads `store.Event` records only; oracle slog events
never land in the store, and `KindOracleCall` journal entries are
never consulted. The only path that produces a full oracle trace today
is `kitsoki export-status --from-trace run.jsonl --journal db` — which
requires both a JSONL trace file (slog records) and the SQLite journal
from the same run.

## 2. Why cassettes are the right place to fix this

The cassette apparatus already solves the "deterministic replay across
multiple calls in declared order" problem and is the agreed substrate
for fixture-backed flows. Layering oracle-trace fidelity on top means:

- **One apparatus, two outputs.** Each episode that records a
  `host.oracle.*` call captures *both* the substituted result (for the
  orchestrator) and the rich metadata (for the trace). Authors don't
  reason about two systems.
- **Cost amortized once.** Real LLM calls cost money. The cassette
  pattern is record-once-replay-forever; the rich metadata is captured
  at the same instant we already pay for the call. There's no separate
  re-record step for trace data.
- **`!include` extends naturally.** Long oracle responses can live in
  sibling JSON files referenced by `!include`, keeping cassettes
  readable.
- **Diff/lint already exists.** `kitsoki cassette diff/lint` becomes
  the diff/lint for oracle traces too, with no new tooling.

## 3. Design

### 3.1 Episode shape — new optional `oracle:` block

Extend `CassetteEpisode` in `internal/testrunner/cassette.go`:

```go
type CassetteEpisode struct {
    ID       string            `yaml:"id"`
    Match    map[string]any    `yaml:"match"`
    Response CassetteResponse  `yaml:"response,omitempty"`
    Delay    string            `yaml:"delay,omitempty"`
    Replay   string            `yaml:"replay,omitempty"`
    Oracle   *EpisodeOracle    `yaml:"oracle,omitempty"`  // NEW

    played bool
}

// EpisodeOracle mirrors journal.OracleCallBody. Present when the
// episode was recorded against a host.oracle.* handler. Replay writes
// this verbatim to the in-memory journal as a KindOracleCall entry.
type EpisodeOracle struct {
    Verb            string          `yaml:"verb"`               // ask | decide | extract | task | converse
    Agent           string          `yaml:"agent,omitempty"`
    Model           string          `yaml:"model,omitempty"`
    DurationMs      int64           `yaml:"duration_ms,omitempty"`
    PromptTokens    int             `yaml:"prompt_tokens,omitempty"`
    ResponseTokens  int             `yaml:"response_tokens,omitempty"`
    CostUSD         float64         `yaml:"cost_usd,omitempty"`
    SystemPrompt    string          `yaml:"system_prompt,omitempty"`
    Prompt          string          `yaml:"prompt,omitempty"`
    Input           json.RawMessage `yaml:"input,omitempty"`
    Response        string          `yaml:"response,omitempty"`
    Error           string          `yaml:"error,omitempty"`
    CallID          string          `yaml:"call_id,omitempty"`
}
```

`!include` works on any value, so long prompts and responses naturally
go to sidecar files:

```yaml
- id: phase_1_repro_oracle
  match:
    handler: host.oracle.task
    phase:   phase_1
  response:
    data:
      submitted: !include 01-repro-report.submitted.json
  oracle:
    verb:          task
    agent:         bugfix-reproducer
    model:         claude-opus-4-7
    duration_ms:   18432
    system_prompt: !include 01-repro-report.system.txt
    prompt:        !include 01-repro-report.prompt.txt
    response:      !include 01-repro-report.response.txt
    call_id:       3e8a-...
```

`oracle:` is purely additive and optional. Episodes for non-oracle
handlers (`host.git`, `host.transport.post`, etc.) leave it unset.
Existing cassettes (e.g. `scout_and_pay.cassette.yaml`) require no
changes.

### 3.2 Record path — capture oracle metadata alongside result

When the dispatcher is in record mode and the live handler is a
`host.oracle.*` name:

1. The fallback handler runs as today; its `host.Result` is appended
   to the cassette as `response`.
2. The dispatcher additionally reads the `KindOracleCall` body the
   handler just wrote to the in-memory journal, keyed by the `call_id`
   the handler returned in its result envelope (the oracle handlers
   already include `call_id` in the result).
3. That body is serialized into the episode's `oracle:` block.

Implementation point: the test rig's `host.Registry` already wraps
each call so the dispatcher knows when a call begins/ends; the journal
write happens inside the handler before return. Reading the journal
entry by `call_id` after return is a single SQL query (or in-memory
lookup if the rig keeps the writer in-process).

The dispatcher's existing per-episode YAML serializer is extended to
emit the `oracle:` block. No change to the matcher.

### 3.3 Replay path — write `KindOracleCall` to the journal

When the dispatcher plays an episode whose `oracle:` block is present:

1. Return `Result{Data: episode.Response.Data}` to the orchestrator as
   today.
2. Additionally, construct a `journal.OracleCallBody` from the
   episode's `oracle:` block and write it to the rig's in-memory
   journal via the existing `appendOracleCallJournal` writer (or a
   thin wrapper that takes the body directly).
3. Optionally emit the `oracle.<verb>.start` and
   `oracle.<verb>.complete` slog records into a capture buffer the rig
   exposes — *or* synthesize them at snapshot-build time from the
   journal (see §3.4). Latter is simpler and avoids ordering issues
   during the call; we prefer it.

The dispatcher needs access to the journal writer. Today the rig owns
the journal; thread it into the dispatcher via `flows.go`'s
`buildOrchestratorRig`.

### 3.4 Snapshot path — synthesize oracle events in `FromHistory`

Extend `runstatus.FromHistory` to accept an optional journal reader
and produce `oracle.<verb>.start/.complete` trace events from
`KindOracleCall` entries:

```go
func FromHistory(
    hist store.History,
    def *app.AppDef,
    sessionID string,
    opts ...HistoryOption,    // NEW: WithOracleJournal(db), etc.
) (Snapshot, error)
```

When `WithOracleJournal` is supplied:

1. Load all `KindOracleCall` entries for `sessionID` (existing
   `journal.LoadOracleCalls` helper).
2. For each entry, generate two `TraceEvent`s:
   - `oracle.<verb>.start` — timestamp = entry timestamp -
     `duration_ms`, attrs = `{verb, agent, model, call_id,
     prompt_preview}`.
   - `oracle.<verb>.complete` — timestamp = entry timestamp, attrs =
     full `OracleCallBody` merged in (mirrors what
     `export_status.mergeOracleBodyIntoAttrs` does today).
3. Merge-sort the synthesized events into the existing store-derived
   event stream by timestamp.

`state_path` for the synthesized events: copy from the
nearest-preceding store event (same approach `mapHistory` uses for
existing events).

The existing `mergeOracleBodyIntoAttrs` and `isOracleCompleteMsg`
helpers in `cmd/kitsoki/export_status.go` move into the `runstatus`
package so both `fromhistory` (synthesize-then-enrich) and
`export_status` (enrich existing) share them.

### 3.5 `fromflow` wiring

The fromflow exporter
(`tools/runstatus/fixtures/cmd/fromflow/main.go`) currently:

```go
hist, err := st.LoadHistory(sid)
snap, err := runstatus.FromHistory(hist, def, string(sid))
```

becomes:

```go
hist, err := st.LoadHistory(sid)
snap, err := runstatus.FromHistory(hist, def, string(sid),
    runstatus.WithOracleJournal(rigJournalDB))
```

`rigJournalDB` is exposed from the testrunner rig via a new field on
the close callback signature, or by reusing the existing journal
handle the rig already creates.

### 3.6 `fromsession` wiring

`fromsession` already takes a `--db` flag pointing at the kitsoki
SQLite database, which holds both the store events and the journal.
Pass the same `*sql.DB` to `WithOracleJournal`. Snapshots from real
recorded sessions then carry oracle events automatically — closing a
pre-existing fidelity gap unrelated to cassettes.

### 3.7 Backwards compatibility

- Episodes without `oracle:` behave identically to today. All existing
  cassettes (`scout_and_pay.cassette.yaml`) pass unchanged.
- `FromHistory` without `WithOracleJournal` behaves identically to
  today. The fromflow/fromsession exporters opt in.
- `mergeOracleBodyIntoAttrs` and `isOracleCompleteMsg` move from
  `cmd/kitsoki/` to `internal/runstatus/` — `export_status.go` updates
  its imports. Pure refactor; no behavior change.

## 4. Migration — bugfix as the first cassette with oracle metadata

1. Replace `stories/bugfix/flows/happy_human.yaml`'s `host_handlers:`
   block with `host_cassette: cassettes/happy_human.cassette.yaml`.
2. Run once in record mode against the real LLM:

   ```sh
   KITSOKI_CASSETTE_RECORD=new_episodes go run ./cmd/kitsoki test flows \
       stories/bugfix/app.yaml --only happy_human
   ```

   This populates the cassette with both `response.data` and `oracle:`
   blocks for every `host.oracle.*` episode, plus result-only episodes
   for non-oracle handlers (git, ci, transport).
3. Long prompts/responses get extracted to sidecar files via
   `!include` for readability (manual or scripted post-record pass).
4. Regenerate the snapshot:

   ```sh
   go run ./tools/runstatus/fixtures/cmd/fromflow \
       --app  stories/bugfix/app.yaml \
       --flow stories/bugfix/flows/happy_human.yaml \
       -o     tools/runstatus/fixtures/bugfix.snapshot.json
   ```

5. Rebuild `bugfix.html` and verify in the Playwright suite that
   `oracle.task.complete` events appear in the timeline and the
   `OracleDetail` sub-renderer shows non-empty prompt/response panes.

Cost estimate: one full happy_human walk through bugfix is ~5 oracle
calls (one per checkpoint phase) at maybe ~$0.50 total — paid once,
captured forever.

## 6. Open questions

1. **Record-time journal access.** The simplest path is for the rig
   to expose its journal `*sql.DB` to the dispatcher at construction
   time. Confirm that's acceptable (it crosses a layer boundary
   `testrunner → journal`, but the rig already imports `journal` for
   the in-memory writer).
2. **Tool-call episodes for `host.oracle.task`.** A real task can fan
   out to many tool calls (read/write/bash/etc) recorded as
   `KindTaskTool` entries. The current proposal does not surface those
   in the cassette — they remain in the journal and are aggregated at
   snapshot-build time by `cmd/kitsoki/export_status.go`'s
   `aggregateTaskDetails`. We should decide whether to also capture
   `KindTaskTool` / `KindTaskEnd` entries in the episode or pull them
   from the journal at replay time the same way real sessions do.
   Probably the latter (the journal write covers it for free if the
   rig writes them during replay), but worth confirming.
3. **`replay: any` semantics for oracle episodes.** A `replay: any`
   episode with an `oracle:` block would write the same
   `KindOracleCall` entry on every play, producing duplicate journal
   rows. Forbid `replay: any` + `oracle:` at load time, or rewrite
   `call_id` per play. Default: forbid; oracle episodes are
   sequential by nature.

## 7. `call_id` is deterministic

Real engine `call_id`s today are per-call UUIDs (`uuid.NewString()`
inside each oracle handler). For cassette-replayed flows we override
this: `call_id` is derived from the episode identity so re-recording
the same flow yields byte-identical journal entries and snapshots.

**Scheme.** On record, the dispatcher computes:

```
call_id = sha256("oracle-call:" + cassette.AppID + ":" + episode.ID)[:16]
```

…and writes that value both into the episode's `oracle.call_id` field
and into the `KindOracleCall` journal entry it writes for that record
(overriding the UUID the handler generated). On replay, the same
derivation is recomputed and used to write the journal entry; the
episode's stored `oracle.call_id` is treated as advisory (re-derived
on every load so a hand-edited episode id rolls forward cleanly).

**Properties.**

- **Stable across re-records.** Episode ids are authored and don't
  change when prompts/responses do, so re-recording the same flow
  produces the same `call_id` and the YAML diff is confined to the
  fields that actually changed.
- **Stable across cassettes for the same app.** Two cassettes for
  `bugfix` with an episode named `phase_1_repro_oracle` get the same
  `call_id` for that episode. Intentional — it lets the runstatus UI
  and downstream tools correlate calls by episode identity.
- **Collision-resistant within and across apps.** 16 hex chars (64
  bits) of SHA-256 is ample for fixture-scale collision avoidance,
  and the app-id prefix prevents cross-app aliasing.
- **No coordination needed.** The dispatcher computes it locally; no
  registry, no counter, no clock dependency.

**Edge cases.**

- *Renaming an episode id* changes `call_id` and produces a diff.
  Acceptable — episode ids are part of the cassette's contract
  (they're shown in `ErrCassetteMiss` and `cassette diff` output).
- *Two episodes with the same id* is already a load-time error in the
  cassette loader (duplicate-id check in `lint`); enforce the same
  invariant at record time so the derivation stays injective.
- *Real (non-cassette) runs* keep their UUID `call_id`s — no change to
  the real engine. Only the replay/record paths inside the testrunner
  use the derived form.

This goes in Phase 3 of §5 (dispatcher journal capture on record) and
Phase 2 (dispatcher journal write on replay).
